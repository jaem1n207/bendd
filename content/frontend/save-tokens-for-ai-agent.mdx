[이전 글](/article/fix-compacting-conversation)에서는 "Compacting conversation..." 이후 답변 품질이 떨어지는 **원인을 분석**하고, 훅 시스템으로 **상태를 보존·복원하는 방법**을 다뤘어요.

이 글에서는 한 단계 앞으로 가요. **Compacting이 오기 전에 토큰을 아끼는 구조적 방법**을 다뤄요.

읽고 나면 다음을 이해할 수 있어요:

- 토큰 비용을 **항목별로 파악**하는 방법
- CLAUDE.md를 **백과사전이 아니라 목차**로 쓰는 이유
- Skill을 **Facade 패턴**으로 설계해서 시스템 프롬프트를 가볍게 유지하는 방법
- 판단이 필요 없는 로직을 **스크립트로 분리**하는 패턴
- Sub-agent로 **컨텍스트를 격리**하는 전략

<Callout type="info">
이 글은 **Claude Code**를 중심으로 설명해요. 지침 파일 최적화, 스크립트 분리 같은 일반 원칙은 다른 AI 코딩 에이전트에도 응용할 수 있지만, Skill이나 Sub-agent 같은 구체적인 기능은 Claude Code 고유의 개념이에요.
</Callout>

## 토큰은 메모리다

Anthropic의 [Claude Code 모범 사례](https://code.claude.com/docs/en/best-practices#be-specific) 문서는 이렇게 말해요:

> Context window fills up fast, and performance degrades as it fills.

전통적인 서버에서는 RAM을 걱정했어요. AI 에이전트에서는 **토큰**을 걱정해야 해요.

| 전통 서버 | AI 에이전트 |
|---|---|
| RAM | Context Window |
| 메모리 점유율 | 토큰 사용률 |
| OOM (Out of Memory) | Compacting (대화 압축) |
| 메모리 릭 | 불필요한 컨텍스트 누적 |

CLAUDE.md, Skills 메타데이터, 대화 히스토리, MCP 응답... 이 모든 것이 Context Window에 쌓여요. 200K 토큰이 많아 보여도, 대규모 코드베이스를 다루다 보면 순식간에 차올라요.

Anthropic Labs의 [Harness 설계 글](https://www.anthropic.com/engineering/harness-design-for-long-running-application-development)에서는 컨텍스트가 차오르면 모델이 **"context anxiety"** 를 보인다고 해요. 작업이 남아 있는데도 컨텍스트 한계에 가까워졌다고 판단해서 성급하게 마무리하려는 경향이에요. 토큰을 아끼면 이런 현상도 늦출 수 있어요.

### 어디서 토큰이 새는가

대략적인 토큰 비용을 알아두면 토큰 예산을 세울 때 도움이 돼요:

| 항목 | 대략적 토큰 | 비고 |
|---|---|---|
| 잘 정리된 CLAUDE.md | 500~2,000 | 프로젝트당 고정 비용 |
| Skill 메타데이터 (개당) | 50~200 | Skill 수 × 단가 |
| 시스템 프롬프트 전체 | 3,000~10,000 | CLAUDE.md + Skills + 기본 지시 |
| 파일 읽기 (500줄) | ~2,000 | 도구 호출 1회 |
| grep 결과 (50 매치) | ~1,000 | 도구 호출 1회 |
| MCP 응답 (DB 쿼리) | 가변 | 큰 응답 주의 |
| 코드 수정 1회 | ~500 | Edit 도구 호출 |

정확한 토큰 수를 계산할 필요는 없어요. **파일 수와 라인 수만 대략 파악하면 충분**해요.

<Callout type="warning">
CLAUDE.md에 "모든 테스트 파일을 분석하라"고 쓰기 전에, 테스트가 50개일 때 어떤 일이 일어날지 상상해 보세요. 50개 × 100줄 = 5,000줄 ≈ 20,000 토큰. 한 번의 지시로 컨텍스트의 10%가 날아가요.
</Callout>

OOM을 예방하듯, 토큰 폭발도 미리 감지할 수 있어요. 지침을 작성한 뒤 Claude에게 "이 워크플로우를 실행하면 어떤 파일들을 읽게 될 것 같아?"라고 물어보세요. 예상보다 많은 파일이 나온다면, **지침을 좁히거나 단계를 나눠야 한다는 신호**예요.

## CLAUDE.md: 백과사전이 아니라 목차

### 거대한 지침 파일의 문제

OpenAI의 Codex 팀은 [Harness Engineering](https://openai.com/index/harness-engineering/) 글에서 "하나의 거대한 AGENTS.md"가 왜 실패하는지 네 가지 이유를 밝혔어요:

1. **컨텍스트는 희소 리소스.** 거대한 지침 파일은 작업, 코드, 관련 문서를 복잡하게 만들어서 에이전트가 핵심 제약조건을 놓치기 시작해요.
2. **지침이 너무 많으면 지침이 아님.** 모든 것이 "중요"하다면 중요한 것은 아무것도 없어요. 에이전트는 의도적으로 탐색하는 대신 패턴 매칭을 수행하게 돼요.
3. **순식간에 망가짐.** 획일적인 거대 매뉴얼은 낡은 규칙들의 무덤이 돼요. 에이전트는 무엇이 여전히 사실인지 알 수 없어요.
4. **확인하기 어려움.** 단일 블롭은 기계적 점검(커버리지, 신선도, 교차 연결)에 적합하지 않아서 드리프트가 불가피해요.

이건 에이전트만의 문제가 아니에요. 소프트웨어 엔지니어링에서도 똑같은 패턴을 봐왔죠. 3000줄짜리 God Class가 왜 문제인지 아는 것처럼, 500줄짜리 CLAUDE.md도 같은 이유로 문제예요.

Anthropic도 [공식 문서](https://code.claude.com/docs/en/memory#claude-md)에서 CLAUDE.md를 **200줄 이하**로 유지하길 권장해요. 파일이 길수록 컨텍스트를 더 많이 소비하고, 지시 준수율(adherence)이 떨어진다고요.

### 목차 패턴: 지침 파일 → docs/

OpenAI Codex 팀의 해법은 단순해요. **AGENTS.md를 백과사전이 아니라 목차로 쓰는 것**이에요.

```md title="안티패턴: 500줄짜리 지침 파일"
CLAUDE.md
├── 기술 스택 설명 (50줄)
├── 코딩 컨벤션 (100줄)
├── 23개 안티패턴 상세 설명 (200줄)
├── 빌드/배포 가이드 (50줄)
├── 보안 규칙 (30줄)
└── 커밋 메시지 규칙 (20줄)
→ 매 대화마다 ~2,000 토큰 소비
```

```md title="권장 패턴: 짧은 지침 파일 + docs/"
CLAUDE.md (50줄, ~200 토큰)
├── @AGENTS.md        → 핵심 규칙 + 구조 맵
├── @docs/pitfalls.md → 필요할 때만 참조
└── @docs/conventions.md → 필요할 때만 참조

docs/
├── architecture.md
├── conventions.md
├── pitfalls.md
├── decisions.md
└── commands.md
```

OpenAI Codex 팀이 실제로 사용하는 구조를 보면 이 패턴이 명확해요. 짧은 `AGENTS.md`(약 100줄)가 목차 역할을 하고, 실제 지식은 `docs/` 디렉토리에 분산돼 있어요:

```md title="OpenAI Codex 팀의 docs/ 디렉토리 구조"
docs/
├── design-docs/
│   ├── index.md
│   └── core-beliefs.md
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── product-specs/
│   └── index.md
├── references/
│   ├── design-system-reference-llms.txt
│   └── uv-llms.txt
├── FRONTEND.md
├── SECURITY.md
└── RELIABILITY.md
```

`@` 참조는 Claude Code가 해당 파일을 **필요할 때** 컨텍스트에 로드하게 해줘요. 매번 전체를 인라인으로 삽입하는 게 아니라, 관련 작업을 할 때만 참조하는 거예요. 최대 5단계 재귀 임포트를 지원하므로, `docs/` 안의 문서가 다시 다른 문서를 참조하는 것도 가능해요.

Claude Code는 [`.claude/rules/` 디렉토리](https://code.claude.com/docs/en/memory#rules)도 지원해요. 여기에 파일 경로별로 규칙을 scope 지정하면, **해당 파일을 작업할 때만** 규칙이 로딩돼요. 모든 규칙을 CLAUDE.md에 넣는 것보다 훨씬 효율적이죠.

OpenAI Codex 팀은 여기서 한 발 더 나가요. 리포지터리의 지식 베이스를 **기록 시스템(System of Record)** 으로 취급하고, 전용 린터와 CI 작업으로 문서가 최신 상태인지 기계적으로 검증해요. 심지어 "doc-gardening" 에이전트가 주기적으로 실제 코드와 맞지 않는 문서를 찾아 수정 PR을 올려요.

<Callout type="info">
CLAUDE.md를 자주 수정하고 있다면, 그 내용은 거기 있으면 안 되는 것일 가능성이 높아요. 동적으로 변하는 정보(현재 작업 이슈, 오늘의 우선순위 등)는 대화로 전달하거나, 별도 파일에 두세요.
</Callout>

### `@` 참조의 토큰 비용

`@` 참조가 실제로 토큰을 얼마나 절약하는지 비교해 볼게요:

| 방식 | 매 대화 고정 비용 | 작업 시 추가 비용 |
|---|---|---|
| 인라인 (500줄 CLAUDE.md) | ~2,000 토큰 | 0 |
| `@` 참조 (10줄 CLAUDE.md + docs/) | ~50 토큰 | 필요한 파일만 ~500 토큰 |

10번의 대화 중 `docs/pitfalls.md`가 필요한 건 2~3번이에요. 인라인이면 10번 모두 2,000 토큰을 쓰고, `@` 참조면 평균 350 토큰이에요. **대화가 쌓일수록 차이는 더 벌어져요.**

## Skill 설계: Facade와 Progressive Disclosure

### Skill 폭발 안티패턴

Claude Code는 시작 시 모든 Skill의 메타데이터(이름, 설명)를 시스템 프롬프트에 로드해요. Skill이 20개면 20개의 설명이 **항상** Context를 점유해요.

```md title="안티패턴: Skill 폭발"
.claude/skills/
├── review-naming/      → 설명 100 토큰
├── review-types/       → 설명 100 토큰
├── review-complexity/  → 설명 100 토큰
├── review-security/    → 설명 100 토큰
├── review-performance/ → 설명 100 토큰
└── ... (15개 더)       → 설명 1,500 토큰
→ 총 2,000 토큰이 매 대화마다 고정 소비
```

이건 전통 아키텍처의 **클래스 폭발(Class Explosion)** 과 같은 안티패턴이에요:

```java title="클래스 폭발 — 매번 어떤 Validator를 써야 하는지 기억해야 함"
class NamingValidator { ... }
class TypeValidator { ... }
class ComplexityValidator { ... }
class SecurityValidator { ... }
// ... 15개 더
```

### Facade 패턴으로 해결

[디미터의 법칙](https://en.wikipedia.org/wiki/Law_of_Demeter)을 떠올려 보세요. Skill 설계에 적용하면: **SKILL.md는 진입점만 제공하고, 세부 지식은 `references/`에 위임**하는 거예요.

```md title="권장: Progressive Disclosure 구조"
.claude/skills/
└── code-review/
    ├── SKILL.md              # "코드 리뷰해줘" → 여기만 로드 (~100 토큰)
    ├── references/           # 필요할 때만 로드
    │   ├── naming-rules.md   # "네이밍 컨벤션은?" → 그때 로드
    │   ├── security-checklist.md
    │   └── performance-guide.md
    └── scripts/
        └── lint-check.sh     # 판단 불필요한 검사는 스크립트로
```

이건 Facade 패턴과 정확히 같아요:

```java title="Facade: 진입점은 하나, 내부는 위임"
class CodeReviewer {
    private NamingRules naming;         // 필요할 때만 로드
    private SecurityChecklist security; // 필요할 때만 로드

    public Review review(Code code) {
        if (needsNamingCheck) naming.check(code);
        if (needsSecurityCheck) security.check(code);
    }
}
```

Claude도 이렇게 동작해요. `SKILL.md`가 Facade 역할을 하고, `references/` 안의 파일들은 **Claude가 필요하다고 판단할 때만** 컨텍스트에 로드돼요.

### 언제 Skill을 분리하고, 언제 합치는가

| 상황 | 전통 아키텍처 | Skill 설계 |
|---|---|---|
| 독립적인 워크플로우 | 별도 Service 클래스 | 별도 Skill |
| 같은 도메인의 세부 규칙 | Private 메서드 / 내부 클래스 | `references/` 파일 |
| 재사용 가능한 유틸리티 | 공통 모듈 | `scripts/` 또는 MCP |

핵심 판단 기준: **이 Skill이 매번 시스템 프롬프트에 로드될 가치가 있는가?** 답이 "가끔"이라면 `references/`로 내려야 해요.

## 결정적 로직을 스크립트로 분리

### LLM이 매번 판단할 필요 없는 것

토큰을 아끼는 가장 직관적인 방법은 **LLM이 할 필요 없는 일을 LLM에게 시키지 않는 것**이에요.

```md title="안티패턴: LLM이 커밋 컨벤션을 매번 해석"
# CLAUDE.md에 이런 규칙이 있다면...
커밋 메시지는 conventional commits 형식을 따른다.
타입은 feat, fix, refactor, chore, docs 중 하나.
스코프는 변경된 모듈명. 본문은 한국어로 작성.
제목은 50자 이내, 본문은 72자에서 줄바꿈.
```

이 규칙을 매번 읽고, 이해하고, 적용하는 데 토큰이 쓰여요. 그런데 이건 **판단이 필요 없는 결정적(deterministic) 로직**이에요.

```bash title="권장: 스크립트가 컨벤션을 캡슐화"
#!/bin/bash
# scripts/validate-commit.sh — git commit-msg 훅으로 등록
MSG=$(head -1 "$1")
PATTERN="^(feat|fix|refactor|chore|docs)(\(.+\))?: .{1,50}$"
if ! echo "$MSG" | grep -qE "$PATTERN"; then
  echo "커밋 메시지가 컨벤션에 맞지 않습니다."
  echo "  형식: type(scope): 한국어 설명 (50자 이내)"
  exit 1
fi
```

```md title="CLAUDE.md에는 스크립트 사용법만"
커밋 메시지 검증: git commit-msg 훅이 자동으로 실행됨
```

LLM 입장에서는 커밋하면 훅이 알아서 검증해요. **규칙 5줄이 스크립트 1줄로 줄어들어요.**

### 스크립트로 분리할 수 있는 것들

| 작업 | CLAUDE.md 규칙 (토큰 소비) | 스크립트 (토큰 절약) |
|---|---|---|
| 커밋 메시지 검증 | 컨벤션 설명 5줄 | commitlint 또는 커스텀 훅 |
| 린트 + 포매팅 | ESLint, Prettier 규칙 나열 | `pnpm lint:fix && pnpm format` |
| PR 템플릿 생성 | 템플릿 형식 설명 10줄 | `./scripts/create-pr.sh` |
| 테스트 실행 + 결과 요약 | 실행 방법 + 결과 해석 규칙 | `./scripts/test-summary.sh` |
| 환경 변수 검증 | 필수 변수 목록 설명 | `./scripts/check-env.sh` |

판단 기준은 단순해요: **입력이 같으면 출력이 항상 같은가?** 그렇다면 스크립트가 LLM보다 나아요. 더 빠르고, 더 정확하고, 토큰을 0개 써요.

<Callout type="info">
Anthropic의 [Claude Code 모범 사례](https://code.claude.com/docs/en/best-practices#be-specific)에서도 CLAUDE.md에 빌드/테스트 명령어를 간결하게 적되, 복잡한 워크플로우는 스크립트로 캡슐화하길 권장해요.
</Callout>

## Sub-agent로 컨텍스트 격리

### 메인 컨텍스트를 보호하세요

Sub-agent는 별도의 컨텍스트 윈도우에서 동작해요. 전통 아키텍처의 **스레드 격리**와 비슷하죠. 메인 스레드에서 무거운 작업을 하면 UI가 멈추듯, 메인 컨텍스트에서 대규모 탐색을 하면 컨텍스트가 넘쳐요.

```md title="안티패턴: 메인 컨텍스트에서 대규모 탐색"
사용자: "이 프로젝트의 모든 API 엔드포인트를 찾아서 문서화해줘"

→ Claude가 직접 수십 개 파일을 읽음
→ 메인 컨텍스트에 수만 토큰 누적
→ Compacting 트리거
```

```md title="권장: Sub-agent에게 위임"
사용자: "이 프로젝트의 모든 API 엔드포인트를 찾아서 문서화해줘"

→ Explore Sub-agent가 파일 탐색 (별도 컨텍스트)
→ 결과 요약만 메인 컨텍스트로 반환 (~500 토큰)
→ 메인 컨텍스트는 깨끗하게 유지
```

Sub-agent가 50개 파일을 읽어도 메인 컨텍스트에는 **요약 결과만** 들어와요. 50,000 토큰이 500 토큰으로 줄어드는 셈이죠.

### 언제 Sub-agent를 쓸지 판단하기

| 상황 | 직접 처리 | Sub-agent 위임 |
|---|---|---|
| 특정 파일 1~2개 수정 | ✅ | ❌ 오버헤드 |
| 코드베이스 전체 탐색 | ❌ 컨텍스트 폭발 | ✅ |
| 독립적인 병렬 작업 | ❌ 순차 처리 | ✅ 병렬 실행 |
| 리팩토링 영향 분석 | ❌ 파일 수 예측 불가 | ✅ |
| 간단한 질문/설명 | ✅ | ❌ 오버헤드 |

Claude Code에서는 [Task 도구](https://code.claude.com/docs/en/sub-agents)로 Sub-agent를 생성할 수 있어요. 각 Sub-agent는 독립적인 컨텍스트를 가지므로, **무거운 탐색 작업을 메인 컨텍스트 밖으로 밀어내는 용도**로 활용하면 토큰을 크게 절약할 수 있어요.

비용도 줄일 수 있어요. 탐색용 Sub-agent에 `model: "haiku"`를 지정하면 **더 빠르고 저렴한 모델**로 읽기 전용 작업을 처리해요. Claude Code의 내장 `Explore` Sub-agent도 기본적으로 Haiku 모델을 사용해요.

<Callout type="warning">
Sub-agent에게 위임한 작업을 메인 컨텍스트에서 다시 하지 마세요. 같은 파일을 두 번 읽으면 토큰이 두 배로 들어요.
</Callout>

## MCP 응답 최적화

MCP(Model Context Protocol)로 외부 시스템에 연결하면 예상치 못한 토큰 폭발이 일어날 수 있어요. DB 쿼리 하나가 10,000행을 반환하면 그대로 컨텍스트에 쌓이거든요.

### 필터링과 페이지네이션

MCP 서버를 설계할 때는 응답 크기를 제한하세요:

```typescript title="안티패턴: 전체 결과를 그대로 반환"
// MCP 도구 핸들러
async function listUsers() {
  const users = await db.query('SELECT * FROM users'); // [!code error]
  return users; // 10,000행 → ~40,000 토큰
}
```

```typescript title="권장: 필터링 + 페이지네이션"
// MCP 도구 핸들러
async function listUsers({ limit = 20, offset = 0, filter }) { // [!code highlight]
  const users = await db.query(
    'SELECT id, name, email FROM users WHERE name LIKE ? LIMIT ? OFFSET ?', // [!code highlight]
    [`%${filter}%`, limit, offset]
  );
  const total = await db.count('users');
  return { users, total, hasMore: offset + limit < total };
}
```

### 응답 형식 최적화

같은 데이터라도 형식에 따라 토큰 차이가 커요:

```json title="비효율적: JSON의 키 이름이 매 행마다 반복"
[
  { "id": 1, "firstName": "Kim", "lastName": "Jaemin", "email": "jaemin@example.com", "createdAt": "2024-01-01" },
  { "id": 2, "firstName": "Lee", "lastName": "Yeongyu", "email": "yeongyu@example.com", "createdAt": "2024-01-02" }
]
```

```md title="효율적: 테이블 형식으로 키 반복 제거"
| id | name | email |
|----|------|-------|
| 1 | Kim Jaemin | jaemin@example.com |
| 2 | Lee Yeongyu | yeongyu@example.com |
```

LLM은 테이블을 잘 이해해요. JSON에서 `"firstName"`, `"lastName"`, `"email"` 같은 키가 매 행마다 반복되는 오버헤드를 테이블로 줄일 수 있어요.

### 도구 설명도 컨텍스트다

MCP 서버가 제공하는 **도구 설명(description)** 도 시스템 프롬프트에 로드돼요. 도구가 많을수록 설명이 쌓여요.

Cursor 팀은 이 문제를 [Dynamic Context Discovery](https://cursor.com/blog/dynamic-context-discovery)로 해결했어요. 모든 도구의 상세 설명을 미리 로드하는 대신, **도구 이름만 포함된 소량의 정적 컨텍스트**를 제공하고 필요할 때 상세 설명을 조회하는 방식이에요. A/B 테스트 결과, MCP 도구를 호출한 실행에서 **총 에이전트 토큰이 46.9% 감소**했어요.

MCP 서버를 직접 만든다면, 도구 설명을 간결하게 유지하세요. 긴 설명은 별도 리소스로 분리하고, 에이전트가 필요할 때 조회하게 하는 게 효율적이에요.

## 대화 습관으로 토큰 아끼기

도구와 설정을 최적화한 뒤에도, **사용자의 대화 습관**이 토큰 효율에 큰 영향을 줘요.

### 구체적으로 지시하세요

```md title="비효율적: 모호한 지시 → 탐색 비용 발생"
이 프로젝트에서 날짜 관련 유틸 함수를 찾아서 고쳐줘

→ Claude가 grep, glob으로 파일 탐색 (~5,000 토큰)
→ 여러 후보 중 맥락 파악 (~3,000 토큰)
→ 수정 (~500 토큰)
```

```md title="효율적: 경로 직접 지정 → 탐색 비용 제거"
src/lib/date-utils.ts:42의 formatDate 함수에서
KST 시간대가 적용 안 되는 문제 수정해줘

→ 파일 읽기 (~1,000 토큰)
→ 수정 (~500 토큰)
```

**탐색 비용을 사용자가 미리 줄여주면** Claude는 핵심 작업에만 토큰을 쓸 수 있어요.

### 세션을 적절히 나누세요

하나의 긴 세션에서 모든 걸 하려 하지 마세요. 컨텍스트가 쌓일수록 주의력은 분산되고, 이전 대화를 참조하는 데 토큰이 소비돼요.

| 작업 성격 | 권장 |
|---|---|
| 서로 관련 없는 버그 2개 | 세션 분리 |
| 하나의 기능 구현 (설계→구현→테스트) | 같은 세션 |
| 큰 리팩토링 + 새 기능 | 세션 분리 |
| 코드 리뷰 피드백 반영 | 같은 세션 |

Anthropic Labs의 [Harness 설계 글](https://www.anthropic.com/engineering/harness-design-for-long-running-application-development)은 긴 작업에서 compaction(대화 압축)보다 **context reset**(컨텍스트 초기화 + 구조화된 핸드오프)이 더 효과적인 경우가 있다고 해요. Compaction은 연속성을 유지하지만 에이전트에게 "깨끗한 시작"을 주지 못해서, context anxiety가 여전히 남을 수 있다는 거예요.

### 컨텍스트를 오염시키지 마세요

Anthropic은 [모범 사례 문서](https://code.claude.com/docs/en/best-practices#avoid-common-anti-patterns)에서 몇 가지 안티패턴에 이름을 붙였어요:

- **Kitchen Sink Session**: 하나의 세션에서 관련 없는 여러 작업을 혼합하는 것. 컨텍스트가 오염되어 모든 작업의 품질이 떨어져요.
- **Infinite Exploration**: 범위를 지정하지 않은 "조사해줘" 요청이 수백 개 파일을 읽어서 컨텍스트를 소진하는 것. Sub-agent로 위임하세요.
- **Correcting Over and Over**: 같은 문제를 2번 이상 교정했다면, `/clear` 후 더 나은 프롬프트로 재시작하는 게 나아요.

작업 사이에 `/clear`를 사용하면 컨텍스트를 초기화할 수 있어요. 간단한 질문은 `/btw`로 하면 대화 히스토리에 추가되지 않아 컨텍스트 성장을 막을 수 있어요.

### `/compact` 타이밍

[이전 글](/article/fix-compacting-conversation)에서 다뤘지만, 다시 한번 강조해요. **자동 압축(100%)이 터지기 전에 `/compact`을 직접 실행**하세요. 체감 70~80%에서 실행하면 `custom_instructions`로 보존 지시를 줄 수 있어요.

```md title="/compact 사용 예시"
/compact TODO 목록, 수정 중인 파일 경로, 현재 작업 맥락을 보존해줘
```

CLAUDE.md에 compaction 지침을 미리 추가해두면, 매번 보존할 항목을 수동으로 지정하지 않아도 돼요.

## 실전 체크리스트

토큰 절약 전략을 한눈에 정리했어요.

**프로젝트 설정 시:**

- CLAUDE.md는 200줄 이하로, 상세 규칙은 `docs/`에 분리
- `@` 참조로 필요할 때만 문서 로드
- Skill은 Facade 패턴으로: 진입점 + `references/`
- 반복적인 컨벤션은 스크립트로 캡슐화
- MCP 응답에 `limit`/`offset` 파라미터 추가

**작업 중:**

- 대규모 탐색은 Sub-agent에게 위임
- 파일 경로와 함수명을 직접 지정
- 관련 없는 작업은 세션을 분리
- 70~80%에서 `/compact` 직접 실행

## 정리

토큰 절약은 성능 최적화와 같아요. 메모리 릭을 고치듯 불필요한 컨텍스트 누적을 줄이고, 캐시를 활용하듯 `@` 참조와 Progressive Disclosure로 필요한 것만 로드하고, 스레드를 분리하듯 Sub-agent로 컨텍스트를 격리하는 거예요.

이번 글의 전략으로 Compacting 빈도를 줄이고, 그래도 Compacting이 터지면 [이전 글의 전략](/article/fix-compacting-conversation)으로 대응하세요. **예방과 대응, 두 가지를 모두 갖추면** AI 코딩 에이전트와의 작업 품질이 확실히 달라질 거예요.
